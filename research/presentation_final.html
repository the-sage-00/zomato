The Question

Can't Zomato already solve this?

Zomato has iterated multiple times — each approach improved but still relies on the same broken signal.

2019-2021
🤖

Pure RL Model

Reinforcement Learning model predicting KPT from order features. No ground truth signal — model only learns from its own mistakes.

❌ Takes 2-3 iterations to converge per restaurant
2022
🔘

FOR Button Introduced

Food Order Ready button added to merchant app. Was supposed to be the ground truth. Improved KPT accuracy by only 9%.

⚠️ If reliable, improvement should be 30-50%
2024-NOW
🔄

Hybrid: FOR + RL

Combines FOR button with per-restaurant RL self-correction. Still takes 2-3 iterations per restaurant to correct.

⚠️ RL exists because FOR signal fails
🔑 Key Insight: Zomato built per-restaurant RL self-correction — you only do that when the base signal fails. The hackathon asks us to "improve signals, not models."
Our Solution

KitchenPulse: 5 Signals > 1 Button

We don't replace Zomato's model. We clean the data that goes into it.

TIER 1 — ₹0 COST

🏆 Corrected Dwell Time

GPS rider wait → subtract parking, walking, handoff noise → pure kitchen wait. Zero-fraud signal.

TIER 1 — VALIDATION

🔍 FOR Trust Score

Don't ignore FOR — score its reliability 0-100. If pressed within 2 min of rider arrival → flagged as rider-triggered.

reliability = valid / total
TIER 2 — BEHAVIORAL

📱 Merchant Behavior

Order accept latency → kitchen load proxy. Slow accept = busy kitchen. Already in Zomato's data.

TIER 2 — AMBIENT

🎙️ AKAI (Ambient Kitchen Activity Index)

TFLite on restaurant tablet measures kitchen noise level. Louder = busier. Leading indicator — catches rush BEFORE dispatch.

TIER 3 — CONTEXT

🌍 External Busyness (Google Popular Times)

Zomato sees 3 orders. Restaurant has 20 dine-in customers. Google data reveals that hidden demand.

🔄 Key Debugging Iterations

Iter 1-3:Hardcoded values, self-referencing, dwell misuse — all KP models worse than baseline
Iter 4:Suspicious threshold too low (3min) → raised to 8min. False positives ↓40%
Iter 6:Archetype priors added — biggest single improvement. MAE: 7.2→6.0
Iter 7:KP built on top of DC, ±3min cap. Final stable architecture achieved.
